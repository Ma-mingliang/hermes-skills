# 报告内容质量检查清单（2026-05-30）

## 使用方法
报告生成后，逐项检查每个板块。未通过的项必须补全。

## 板块检查清单

### 一、Agent生态
- [ ] 全能Agent有版本号+上下文+定价
- [ ] 新Agent有**前辈对比表**（功能/成本/自主性/易用性/适用场景）
- [ ] 新Agent有**GitHub链接**（可点击URL，不是"GitHub"两个字）
- [ ] 新Agent有**原理分析**
- [ ] 组件说明**解决哪些Agent的什么问题**
- [ ] 组件有**适用Agent**列表

### 二、Skills市场
- [ ] 每个Skill有**GitHub链接**
- [ ] 每个Skill有**痛点类型标注**（能力缺失型/使用不便型/...）
- [ ] 每个Skill有**原理分析**（🎯原理+🔧操作+📊效果+👤案例）
- [ ] 每个Skill有**前辈对比**（表格，不是一句话）
- [ ] 每个Skill有**真实使用案例**

### 三、模型动态
- [ ] 模型版本**精确到小版本**（不硬编码，实时获取）
- [ ] 有**综合能力排名**（基于当日数据）
- [ ] 有**编码能力排名**
- [ ] 有**性价比排名**
- [ ] 有**热点模型事件**（HN热度pts）
- [ ] 有**版本更新**列表

### 四、行业热点+行业应用
- [ ] 行业应用覆盖**≥3个行业**
- [ ] 每个事件有**可点击的原文链接**（不是"36氪"两个字）
- [ ] 每个应用说明**解决什么行业问题**
- [ ] 每个应用说明**技术原理**（不是关键词，要展开）
- [ ] 不需要Stars

### 五、MCP动态
- [ ] MCP项目有**GitHub链接**
- [ ] 每个项目有**Stars+功能+适用场景**
- [ ] 有**生态趋势**分析（不是一句话）
- [ ] 有**与Agent/Skills的关联**
- [ ] MCP server**优先放MCP板块**，不放组件

### 六、数据面板
- [ ] 有**信源覆盖**统计
- [ ] 有**分类统计**（Skills/组件/Agent数量）
- [ ] 有**API消耗**统计

### 七、核心信号
- [ ] **3-7条**
- [ ] 每条基于**当日数据**
- [ ] 类型覆盖：趋势/安全/生态/模型/市场/行业

## 推送质量原则

1. **内容质量优先**：宁可4段高质量，不要8段敷衍
2. **不必拼凑**：不限分段限制，按内容自然分段
3. **每段可长**：3000-4000字一段没问题
4. **先完整再分段**：先生成完整报告，再按内容切分推送

## 2026-05-30教训

**错误做法**：为了凑8段，把Skills内容压缩到一句话，缺少GitHub链接、原理分析、前辈对比表。

**正确做法**：先按full版标准生成完整报告（不限字数），再自然分段推送。用户说"不必拼凑，不必考虑分段限制"。
