## A. 数据收集类

### P50: DDG批量采集完全不可用
- **问题**：DuckDuckGo news search第2个查询就403，text search同session返回全0
- **根因**：DDG对同一IP的批量请求限流极严，1.2s间隔也无法避免
- **规则**：DDG最多跑1个查询，然后立即切换到HN Algolia + GitHub API
- **详见**：`source-layers.md` Layer 6

### P49: GitHub API限流处理
- **问题**：14个GitHub查询中4个被403限流
- **规则**：限流时跳过该查询，不阻塞流程；优先查询高权重关键词

### P28: GitHub API date-filtered search两遍策略
- **问题**：窄窗口日期过滤+精准关键词=极稀疏结果
- **规则**：第一遍日期过滤（找本周新项目），第二遍无过滤（找高星项目）

### P36: 数据采集不完整
- **问题**：只采集了5个数据源，预期102个
- **规则**：RSS失败→降级到聚合器；API限流→跳过不阻塞；必须尝试备用源

### P31: Jimmuji聚合器优先策略
- **发现**：直接API采集大面积失败时，Jimmuji的昨日报告提供了90%的实质新闻
- **规则**：Jimmuji不仅是辅助源，而是降级场景下的核心内容支柱

### P4: 中文源访问限制 + SSL间歇性故障
- **规则**：36kr RSS是最可靠的中文源；SSL故障时重试；降级到Jimmuji

### P14: Cron环境下的搜索策略
- **规则**：不要用delegate_task做搜索；用execute_code+terminal(curl)直接调API

## B. 数据验证类

### P48: 数据验证必须区分"今日/本周/历史"
- **问题**：报告没有区分"今日新闻"和"本周高星项目"
- **规则**：HN按created_at分三类；GitHub按created_at分三类；报告用时间分类标签

### P49: 数据验证必须检查实际时间戳
- **问题**：只检查了"是否有数据"，没有检查"数据的时间戳是否正确"
- **规则**：使用verify_data_timestamps()函数，区分今日/本周/历史

### P25: 时间戳计算必须用当日0点
- **问题**：用了24小时前的时间戳，导致数据可能是"近24小时"而不是"今日"
- **规则**：用today.replace(hour=0, minute=0, second=0)的时间戳

### P48: 报告标题必须与数据一致
- **问题**：报告写了"🆕新出现"但实际数据没有新项目
- **规则**：🆕板块只在有真正新项目时才出现；HN热度>100pts才能标注"热点"

## C. 分类类

### P24: 分类流程优先级
- **正确优先级**：MCP > Agent组件 > Agent > Skills
- **流程**：Step 1(含"skill(s)")→Step 2(主要是.md)→Step 3(含"mcp server")→Step 4(含"for ai agents")→Step 5(含"ai agent")→Step 6(以上都不满足→组件)

### P0: 三分类标准
- **Agent** = 可独立运行 + 有API/Web UI
- **Skills** = 纯.md文件集合，不能独立运行
- **Agent组件** = 有代码但不能独立运行，增强现有Agent

### P2.9: 有Python ≠ Agent
- **问题**：看到GitHub仓库有Python代码就归为Agent
- **规则**：必须走分类决策树，不能凭代码语言判断

### P13: 分类-报告一致性
- **问题**：数据采集阶段正确分类了项目，但写报告时凭印象重新分类
- **规则**：报告生成时必须查询分类标签，不能凭搜索来源决定板块

### P44: 分类审查必须执行
- **已知误判**：ECC(component被误判为skill)、Dify(agent_specialized被误判为mcp)
- **规则**：Step 4.5分类审查必须执行，检查已知误判案例

### P48: 已知项目白名单机制
- **问题**：P24决策树对知名项目描述关键词覆盖不足
- **规则**：维护KNOWN_PROJECTS白名单，分类时先查白名单再走决策树

### P51: 专精Agent分类遗漏
- **问题**：专精Agent只有1个，其他被错误分类为component
- **规则**：搜索垂直领域关键词；如果当日没有新专精Agent，推荐5个高星专精Agent

### P52: 专精Agent搜索策略
- **必须搜索的关键词**：DB-GPT、HolmesGPT、TradingAgents、AiSOC、security+agent、data+agent

## D. 报告生成类

### P16: 报告格式必须用emoji编号
- **规则**：用📌/🆕/⭐ emoji标识，不用"第二类/第三类"

### P16.1: 微信推送分段格式
- **规则**：按内容自然分段，无字数限制；对比表列：定位/成本/自主性⭐/易用性⭐/适用人群

### P2.10: Skills也必须前辈对比拆分
- **规则**：每个新Skill必须找到功能相似的高星前辈Skill进行对比

### P2.11: 描述必须通俗易懂
- **规则**：每个Agent/Skills/组件的描述都必须解释这是解决什么问题

### P54: 痛点描述必须按9种问题类型框架
- **9种类型**：能力缺失/使用不便/成本过高/安全风险/效率低下/知识壁垒/协作困难/数据孤岛/行业落地

### P55: Agent必须有具体核心功能描述
- **问题**：CowAgent写"Agent缺乏任务规划"太笼统
- **规则**：必须说明Agent的核心功能是什么、能用来干什么

### P50: 前辈对比必须有详细文字分析
- **规则**：每个新Agent/Skill必须有4段文字分析：相似/不同/解决什么问题/原理差别

### P18: Skills内容详略规则
- **规则**：所有skill不论stars高低，都必须按完整格式写（痛点+原理+案例+前辈对比）

### P19: 前辈对比 ≠ 高星Skills对比
- **规则**：前辈对比放在每个skill下面；高星对比单独板块，不重复

### P53: 行业应用必须覆盖≥3个行业
- **规则**：从多个来源填充，检查行业emoji数量

### P59: 数据来源透明度
- **规则**：每个板块必须标注来源（✅API/⚠️知识/❌未验证）；量化效果无来源必须删除

### P58: 报告中的每个项目必须来自数据收集
- **规则**：不能凭skill记忆直接写；如果数据收集中没有找到，就不能写在报告中

## E. 推送类

### P39: 微信推送限流
- **规则**：每段间隔60秒；限流后等120秒重试；超过2次限流保存本地

### P17: 微信推送限流处理（ret=-2）
- **已验证节奏**：每段间隔15秒可稳定发送8段；10秒间隔可能触发限流

### P50: 微信推送CLI限制
- **规则**：CLI模式下没有send_message工具；微信推送必须通过cron job的final response

### P15: 推送模式
- **规则**：Cron模式用final response；交互模式直接在回复中输出；CLI模式直接输出

## F. 执行纪律类

### P43: 必须严格按照清单逐步执行
- **问题**：跳过了完整流程，直接快速生成简化版报告
- **规则**：必须按Step 1-7逐步执行；不必追求速度，质量优先

### P47: 执行验证循环
- **流程**：执行→验证→通过→下一步；验证失败→修复→重新验证

### P35: 执行完整性问题
- **问题**：只做了"修复"，没有做"验证"
- **规则**：更新skill后必须重新生成报告；重新生成后必须执行质量检查

### P30: execute_code函数隔离陷阱
- **规则**：每个execute_code调用必须自包含，所有依赖函数都在同一个代码块内定义

### P2.7: 子任务搜索结果必须验证
- **问题**：delegate_task子任务返回的数据可能完全编造
- **规则**：GitHub项目→web_fetch验证URL；stars数→GitHub API确认；未验证数据不能写入报告

## G. 技术类

### P46: GitHub API URL编码问题
- **规则**：使用urllib.parse.quote()编码查询字符串

### P49: 数据结构可能是list或dict
- **规则**：使用前先检查类型；统一使用list，统计数据用len()

### P2.6: write_file WSL故障
- **规则**：使用execute_code + Python open().write()直接写入，完全绕过WSL

### P2.9: Python urllib直接请求
- **规则**：WSL/terminal不可用时，用Python urllib.request直接发起HTTP请求

### P41: 模型数据采集工作流程
- **数据源**：lmarena.ai（排名）、OpenRouter API（价格）、artificialanalysis.ai（能力）

### P42: 模型排名网站访问情况
- **结果**：lmarena.ai/artificialanalysis.ai/designarena.ai有Vercel安全检查，无法访问
- **降级**：使用OpenRouter API获取模型列表和价格
