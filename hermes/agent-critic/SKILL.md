# 批判者 (Agent Critic)

批判者是针对AI Agent输出的多维度批判性分析框架。它不预设"输出一定是错的"——而是系统性地审视输出的每一个可质疑之处，让隐藏的问题浮出水面。批判的目标不是否定，是让输出更可靠。

---

## 批判维度总览

批判者从8个维度审视输出，每个维度回答一个核心问题：

| 维度 | 核心问题 |
|------|----------|
| **1. 逻辑严密性** | 推理链是否成立？有没有逻辑跳跃或谬误？ |
| **2. 证据充分性** | 论点有证据支撑吗？证据的质量和相关性如何？ |
| **3. 假设透明性** | 隐藏了哪些前提假设？这些假设可靠吗？ |
| **4. 偏见与盲点** | 是否存在认知偏见、视角局限或利益相关盲区？ |
| **5. 完整性与边界** | 遗漏了什么？边界情况是否处理？适用范围是否被过度扩展？ |
| **6. 实用可行性** | 在实际条件下能否落地？有没有被忽略的实施障碍？ |
| **7. 表达清晰性** | 是否存在歧义、误导性措辞或过度自信的语言？ |
| **8. 校准诚实性** | 表达的确定性是否与实际的证据强度匹配？ |

## 分析流程

### Step 1: 确认分析目标

明确要分析的是什么：
- **目标对象**：哪一段输出？什么类型的输出（推理/方案/代码/决策/解释）？
- **使用场景**：这个输出是用来做什么的？出错的后果有多严重？
- **分析深度**：快速扫描（3-5分钟）还是深度审查（10-15分钟）？

如果是用户直接提供的文本，直接分析。如果上下文中有之前的Agent输出，默认分析最近的那一段。

### Step 2: 快速扫描 —— 粗筛重大问题

在所有维度中做第一遍快速扫描，找出最明显的问题区域。这一步只找"一眼就能看出的问题"：

- 有没有自相矛盾？
- 有没有明显的事实错误？
- 有没有偷换概念或逻辑跳跃？
- 有没有过度推广（从小样本推到普遍结论）？

如果快速扫描发现致命问题，标记后继续；快速扫描干净不代表没有问题，只代表问题更隐蔽。

### Step 3: 逐维度深度分析

对每个维度逐一深入检查。不需要每个维度都写长篇分析——根据输出本身的特点，在有问题的地方深入，没问题的维度简要说明即可。

#### 1. 逻辑严密性

检查推理结构：

- **前提-结论链**：提取主要论证的前提和结论，检查前提是否支持结论
- **逻辑一致性**：输出内部是否有矛盾？前后说法是否一致？
- **常见谬误**：扫描是否有以下模式：
  - 错误二分（只给两个选项，实际上有更多）
  - 稻草人（歪曲某个观点再攻击它）
  - 循环论证（结论偷偷放在了前提里）
  - 滑坡论证（假定一连串未证明的因果链）
  - 因果混淆（相关性当成因果性）
  - 以偏概全（从少数案例推到普遍结论）
  - 诉诸权威（用权威代替证据）
  - 诉诸无知（"没人证明它是错的所以它是对的"）

判断等级：
- ✅ 逻辑成立 — 推理链完整
- ⚠️ 有瑕疵 — 存在可疑的推理步骤，但整体方向没问题
- ❌ 逻辑断裂 — 存在关键推理链不成立或谬误

#### 2. 证据充分性

评估支撑论点的证据：

- **证据存在吗？** — 每个事实性断言是否有来源或依据？
- **证据质量** — 是什么类型的证据？
  - 直接观察 > 可靠推断 > 间接推断 > 个案/轶事 > 未验证假设 > 纯断言
- **证据相关性** — 证据是否真的支持所声称的论点？有没有证据-主张错配？
- **关键缺失** — 如果是真的，我们应该看到什么证据？有没有注意到不该缺失的东西？
- **来源质量** — 如果有引用，来源是否可信？是否有利益冲突？

判断等级：
- ✅ 证据充分 — 每个关键主张都有坚实支撑
- ⚠️ 证据不足 — 部分主张缺乏足够支撑
- ❌ 缺乏证据 — 关键主张无据可依

#### 3. 假设透明性

挖掘未明说的前提：

- **显性假设**：输出中明确说出的假设，它们合理吗？
- **隐性假设**：输出依赖但未说出的前提——"如果要让这个推理成立，什么必须是真的？"
- **假设的脆弱性**：哪个假设一旦不成立，整个输出就会坍塌？
- **应该被质疑的假设**：是否存在被当做事实对待但实际上是假设的陈述？

判断等级：
- ✅ 假设透明 — 关键假设都被识别并讨论
- ⚠️ 假设隐藏 — 存在未明说的前提，但不一定会颠覆结论
- ❌ 假设脆弱 — 结论依赖未经检验的关键假设

#### 4. 偏见与盲点

识别认知偏见和视角局限：

- **确认偏误** — 只提供支持已有结论的证据，未考虑反面证据
- **锚定效应** — 过度依赖某个初始信息
- **可用性启发** — 过度依赖容易想到的例子
- **框架效应** — 问题的表述方式限定了解决空间
- **幸存者偏差** — 只看到成功案例，忽略了失败案例
- **近因效应** — 过度权重近期信息
- **单一视角** — 只从一个角色/学科/文化视角看问题
- **利益相关盲区** — 输出是否在服务某个未被声明的立场？

#### 5. 完整性与边界

检查遗漏和过度扩展：

- **遗漏了什么？** — 有没有该讨论但没讨论的关键问题？
- **边界情况** — 极端输入、空值、并发、失败场景是否被考虑？
- **适用范围声明** — 结论的适用范围是否被明确界定？
- **过度推广** — 是否从特定场景的结论推广到了不应适用的场景？
- **对立观点** — 是否承认并回应了合理的反对意见？

#### 6. 实用可行性

评估在真实世界中的可操作性：

- **资源约束** — 所需的时间、人力、资金是否合理？
- **技术可行性** — 在当前技术条件下能否实现？
- **依赖风险** — 是否依赖未验证的外部条件或第三方？
- **实施路径** — 是否有可操作的第一步？还是停留在抽象描述？
- **失败模式** — 最可能的失败方式是什么？有没有备选方案？
- **隐性成本** — 有哪些容易被忽略的成本（切换成本、学习成本、维护成本）？

#### 7. 表达清晰性

审查语言表达：

- **歧义** — 是否存在多种理解的表述？
- **关键术语定义** — 核心概念是否有明确定义？
- **过度自信的语言** — "显然"、"毫无疑问"、"必然"等措辞是否有足够支撑？
- **模糊量词** — "很多"、"大幅"、"显著"等是否有具体数据对应？
- **情感操纵** — 是否存在使用情感化语言代替理性论证的情况？
- **格式可读性** — 结构是否清晰，关键信息是否容易被找到？

#### 8. 校准诚实性

评估表达的信心是否与证据强度匹配：

- **确定性标记检查** — "一定"、"必然" vs "可能"、"初步看来"——措辞的确定性与证据匹配吗？
- **不确定性量化** — 是否给出了概率/置信度估计？如果有，合理吗？
- **知识边界承认** — 是否诚实承认了未知的领域？
- **过度承诺** — 是否有无法兑现的保证或承诺？

判断等级：
- ✅ 校准良好 — 表达的确定性与证据强度匹配
- ⚠️ 轻微过度自信 — 部分陈述的信心超出证据支撑
- ❌ 严重失准 — 在证据薄弱处使用了高度确定的语言

### Step 4: 综合评估

汇总所有维度的发现，给出整体判断：

1. **致命问题**：是否有使输出根本上不可靠的问题？
2. **重要问题**：是否有显著削弱输出质量的问题？
3. **改进空间**：有哪些可以做得更好的地方？

### Step 5: 生成改进建议

为每个发现的问题提供具体、可操作的改进建议。建议应该告诉作者"怎么做"而不是只说"不对"。

## 特殊模式

### 快速批判模式

当用户说"快速过一下"、"扫一眼"时，只执行 Step 2（快速扫描）和 Step 4（综合评估），跳过逐维度深度分析。输出格式简化为：

```
# 快速批判扫描

## 发现的问题
- [问题1] — [严重程度：致命/重要/轻微]
- [问题2] — [严重程度]

## 整体判断
[一句话总结]

## 最值得改进的一点
[选择最有价值的改进建议]
```

### 深度路由模式

当某个维度发现问题特别严重时，主动建议路由到专项技能：
- 逻辑问题严重 → `/logic-check` 或 `/logic-argument-validation`
- 证据问题突出 → `/investigation-evidence-audit`
- 假设隐藏很深 → `/creativity-assumption-excavator`
- 信心失准 → `/probability-confidence-calibration`
- 偏见明显 → `/ethics-bias-check`
- 整体质疑 → `/epistemology-epistemic-status`

## 运行检查点

使用 AskUserQuestion 工具与用户确认：

- **Question:** "如何运行批判分析？"
- **Header:** "分析范围"
- **Options:**
  - **完整审查** — 全部8个维度深度分析
  - **快速扫描** — 粗筛重大问题，跳过细节
  - **聚焦特定维度** — 只深入分析用户指定的维度
  - **仅输出改进建议** — 跳过分析过程，直接给出可操作的改进清单

根据用户选择执行。

完成分析后：

- **Question:** "分析完成。下一步？"
- **Header:** "后续操作"
- **Options:**
  - **深入某个维度** — 对问题最多的维度执行专项深度分析
  - **重新生成改进版** — 基于批判结果重新输出修正后的版本
  - **路由到专项技能** — 将最严重的问题交给专项技能深入处理
  - **结束** — 完成

## 注意事项

1. **批判 ≠ 否定**：批判的目标是提升质量，不是挑刺。在指出问题的同时，也承认输出做得好的地方。
2. **比例感**：对"可能性的瑕疵"和"确定性的错误"使用不同语气。不要对次要问题使用致命问题的语气。
3. **可操作性**：每条改进建议必须具体到"怎么做"，而不是停留在"这不对"。
4. **自我应用**：你在执行这个skill时也在产生输出——你自身的批判分析也应该经得起同样的审视。
5. **语言尊重**：批判时保持尊重和建设性，不居高临下、不冷嘲热讽。
