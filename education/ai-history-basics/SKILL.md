## AI发展统一理论（GPT时代版）

### 核心观点

AI的发展史，本质上是在解决一个问题：

> 如何让 AI 像人一样完成工作，并最终在规模、速度、成本和能力上超越人类。

```text
人类目标 → AI理解 → AI决策 → AI执行 → 目标完成
```

整个 AI 发展过程，就是不断缩小这条链路中的各种鸿沟（Gap）。

### 真正的发展主线

很多人认为主线是：Transformer → GPT → Agent（技术路线）

实际上真正的发展主线是：**让AI完成越来越复杂的人类工作**

每当 AI 尝试替代人类某部分工作时，就会暴露新的瓶颈，整个行业开始围绕这个瓶颈疯狂优化。

```text
旧问题暴露 → 新方案出现 → 形成热词 → 形成生态 → 继续暴露新问题
```

所以 AI 历史不是技术演进史，而是**问题解决史**。

## 五条主线框架

因为要替代人类工作，必须同时满足五个条件。缺少任何一个条件，AI都无法真正替代人。

### 第一条主线：AI能不能做到（Capability）

核心问题：**AI太笨**

如果 AI 连任务都理解不了，后面一切都没有意义。

| 代际 | 问题 | 解决方案 |
|------|------|----------|
| 第一代 | 不会理解语言 | Transformer, BERT, GPT |
| 第二代 | 知识容量不足 | Scaling Law, GPT-3, Chinchilla |
| 第三代 | 不会推理 | CoT, ToT, Self Consistency, ReAct |
| 第四代 | 推理能力增长放缓 | Reasoning Model, o1, DeepSeek-R1, Test-Time Compute |
| 第五代 | 只能理解文本 | CLIP, Diffusion, Vision-Language, VLA, World Model, Physical AI |
| 第六代 | 能力提升越来越昂贵 | MoE, Mamba, Sparse Model |

本质：**让AI越来越聪明**

---

### 第二条主线：AI做的是不是对的（Reliability）

核心问题：**AI会胡说**

人类员工可能犯错，但不会凭空捏造。大模型会。

| 代际 | 问题 | 解决方案 |
|------|------|----------|
| 第一代 | 输出极不稳定 | Prompt Engineering |
| 第二代 | 不符合人类偏好 | RLHF, Alignment, Constitutional AI |
| 第三代 | 知识幻觉 | Embedding, VectorDB, RAG, GraphRAG |
| 第四代 | 推理过程不可信 | Verifier, Critic, Reflection, Self-Refine |
| 第五代 | 无法证明模型进步 | Harness, HELM, MMLU, HumanEval, SWE-Bench, Arena, LiveBench |

本质：**让AI越来越可信**

---

### 第三条主线：AI能不能替我做（Execution）

核心问题：**AI会说不会做**

GPT-3时代最大的矛盾。

| 代际 | 问题 | 解决方案 |
|------|------|----------|
| 第一代 | 不会操作工具 | Tool Use, Function Calling |
| 第二代 | 不会自主执行任务 | Agent, AutoGPT, BabyAGI |
| 第三代 | 复杂任务无法编排 | Workflow, LangChain, LangGraph, DAG |
| 第四代 | Agent不稳定 | Planner, Executor, Critic, Multi-Agent |
| 第五代 | Workflow无法复用 | Skill, Capability, Plugin（Skill = Workflow产品化） |
| 第六代 | 技能生态碎片化 | MCP（MCP = Agent世界的USB协议） |
| 第七代 | Agent没有长期记忆 | Memory, Mem0, MemoryOS |
| 第八代 | Agent无法操作现实软件 | Browser Use, Computer Use, GUI Agent |
| 第九代 | Agent执行后即消失 | OpenClaw, Persistent Agent |
| 第十代 | Agent不会成长 | Hermes, Learning Loop, Skill Evolution |

本质：**让AI成为数字员工**

---

### 第四条主线：AI值不值得替代人（Cost）

核心问题：**AI太贵**

如果 AI能力=人类10倍，成本=人类100倍，依然无法替代人。

| 代际 | 问题 | 解决方案 |
|------|------|----------|
| 第一代 | 训练太贵 | DeepSpeed, ZeRO |
| 第二代 | 推理太贵 | FlashAttention, vLLM, SGLang |
| 第三代 | 微调太贵 | LoRA, QLoRA |
| 第四代 | 本地部署困难 | Quantization, GPTQ, AWQ, GGUF, Ollama |

本质：**让AI成本低于人类**

---

### 第五条主线：AI到底创造什么价值（Value）

核心问题：**AI再强没人用，依然失败**

所以必须落地。

| 代际 | 价值领域 | 代表 |
|------|----------|------|
| 第一代 | 知识工作 | ChatGPT |
| 第二代 | 设计工作 | Midjourney, Stable Diffusion |
| 第三代 | 搜索工作 | Perplexity |
| 第四代 | 编程工作 | Copilot, Cursor, Claude Code, OpenHands |
| 第五代 | 研究工作 | Deep Research |
| 第六代 | 体力工作 | Physical AI, Robotics |

本质：**把能力转化成生产力**

## 五条主线之间的因果关系

这五条线不是并行分类，而是因果关系：

```text
能力不足 → 提升能力
能力提升后 → 发现不可信
可信后 → 发现不会执行
执行后 → 发现太贵
成本下降后 → 开始大规模创造价值
```

然后价值场景又反向提出新的能力需求，形成闭环。

### 因果链示例（必须掌握）

**示例1：RAG的诞生**
```
GPT-3能力强（能力）→ 用户期望AI知道最新信息 → 知识截止问题暴露（可信度）
→ RAG技术出现（可信度方案）→ 需要向量数据库（执行）→ 需要Embedding模型（能力）
```

**示例2：Agent的诞生**
```
GPT-4推理强（能力）→ 用户期望AI能做事 → 无法执行动作问题暴露（执行）
→ Agent架构出现（执行方案）→ 需要工具调用（执行）→ 需要MCP协议（执行）
→ Agent不稳定 → 需要Multi-Agent（执行）→ 成本上升 → 需要降低成本（成本）
```

**示例3：Vibe Coding的诞生**
```
推理能力提升（能力）→ 编程门槛问题暴露（价值）→ Vibe Coding出现（价值方案）
→ 需要更好的上下文理解（能力）→ 需要Context Engineering（能力+执行）
```

## 最终结论

AI发展的真正主线不是 Transformer、GPT 或 Agent。

真正的主线是：**让AI可靠地替代越来越复杂的人类工作**

而所有热词、本轮技术浪潮和未来的新概念，本质上都只是在解决五个永恒问题：

1. AI能不能做到？（Capability）
2. AI做的是不是对的？（Reliability）
3. AI能不能替我做？（Execution）
4. AI值不值得替代人？（Cost）
5. AI到底创造什么价值？（Value）

这五个问题共同构成了 GPT 时代以及未来 Agent 时代的统一解释框架。

### 进度追踪（必须执行）

**进度文件**：`~/.hermes/ai-history-state.json`
**更新脚本**：`scripts/update_state.py` — 自动处理中文字段名，防止字段漂移
用法：`execute_code` 中调用 `terminal(f'python {skill_dir}/scripts/update_state.py {day} "{topic}"')`

**文件格式**（`update_state.py` 使用中文字段名作为 canonical 格式）：
```json
{
  "current_day": 15,
  "last推送_date": "2026-06-05",
  "total推送": 14,
  "history": [
    {"day": 1, "topic": "什么是人工智能？", "date": "2026-05-22"},
    {"day": 2, "topic": "神经网络的诞生", "date": "2026-05-23"}
  ]
}
```

**Pitfall**: `scripts/update_state.py` 内部使用中文字段名（`last推送_date`, `total推送`）作为 canonical 格式。如果手动编辑状态文件或在 `execute_code` 中直接更新，请保持与脚本一致使用中文字段名，否则脚本下次运行会触发"WARNING"并重新转换。如需在代码中读取状态，注意同时兼容两种字段名。

**Pitfall（state 文件可信度）**：state 文件可能与实际推送进度不同步（cron 失败后手动调整、多次 session 交叉更新、用户手动重置等）。读取 state 后，**必须向用户确认要推送的 Day 号**，不要盲目信任 `current_day`。2026-06-12 session 中 state 显示 Day 3，实际用户要求推送 Day 7——如果直接按 state 推送会推错内容。

**更新规则（必须执行）**：
- 每次推送完成后必须更新进度文件
- current_day 必须 +1
- total推送 必须 +1（注意中文字段名）
- history 必须追加本次记录
- 如果 current_day > 60，必须重置为 1
- 使用 `execute_code` 而非 `write_file`，因为中文字段名在 write_file 中可能编码异常

**WeChat 推送规则**：
- 故事版（2000字+）必须分段推送，每条消息 ≤4000 字符
- 每条消息必须是完整段落，不能在段落中间截断
- 推送顺序：先发正文前半部分 → 中间部分 → 结尾（含讨论问题和延伸阅读）
- `send_message` 必须指定 `target`（如 `weixin`），否则报错
- **严禁缩减内容长度**：用户明确要求"完整版"时，每个故事版必须达到2000字+的完整内容（中文字符数+英文单词数×2），不得为了"精简"而删减人物描写、场景描写、对话、情节转折等要素。用户曾因内容不够完整要求重新推送（2026-06-11）。
- **限流退避**：连续发送多条消息时，微信可能触发 `rate limited: ret=-2`。每次发送后等待至少60秒再发下一条。如果触发限流，等待120秒后重试。批量推送Day 1-6等场景尤其容易触发。
- **批量推送限流**：连续发送多条微信消息时，每发2-3条后必须等待2分钟（120秒），否则会触发 `rate limited: ret=-2`。推送6个Day的完整内容时，预计需要6-8分钟。
- **批量重推流程**：用户要求"重新从Day X到Day Y全部推送"时，必须逐个生成每个Day的完整故事版（2000字+），逐条send_message发送。不可跳过任何Day，不可缩减内容长度。每发2条后暂停等待120秒避免限流。使用 `terminal("sleep 120", background=True, watch_patterns=["ready"])` 或手动计时。2026-06-11 用户要求重推 Day 1-6，首次尝试因限流失败，分批间隔后成功。
- **推送纪律**：用户要求"完整版"时，故事版必须≥2000字（中文字符数），不得偷工减料。用户已明确纠正过"我需要的完整版"——每次推送前必须自检字数。

**Pitfall（Windows cron 环境）**: 在 Windows 主机上运行 cron job 时，`terminal` 工具可能因 WSL NAT 错误失败。优先使用 `execute_code` 中的 `from hermes_tools import terminal` 或直接用 Python `open()`/`json` 操作文件，避免依赖顶层 `terminal` 调用。

## 推送失败手动恢复（必须掌握）

当 cron job 执行成功（state 已更新）但微信投递失败时，需要手动恢复。

### 诊断流程
1. `cronjob list` 检查 last_status 和 last_delivery_error
2. 常见错误模式：
   - `Cannot connect to host ilinkai.weixin.qq.com:443 ssl:default [getaddrinfo failed]` → 微信 DNS 解析失败，Gateway 网络不通
   - `iLink sendmessage rate limited: ret=-2` → 微信限流，需要清除 context_token 并重启全局 Gateway
   - `HTTP 402` → 余额不足，需在 cron job 上指定 model + provider
3. 读取 state 文件确认当前进度

### 恢复步骤
1. 用 `execute_code` + Python `json` 读取 `~/.hermes/ai-history-state.json`
2. **关键陷阱**：state 文件的 current_day 已经 +1，所以要推的是 `history` 数组最后一条记录，不是 current_day
3. 确认要推的 Day 号和 topic
4. 根据版本选择规则生成内容（Day 1-18 = 故事版，Day 19-60 = 学习版）
5. 用 `send_message(target='weixin')` 手动发送
6. **不需要再更新 state 文件**（cron job 已经更新过了）

### Pitfall：state 已更新但未投递
cron job 的执行逻辑是"生成内容 → 更新state → 发送消息"。如果发送失败，state 已经被更新了。恢复时只需重新发送，不要重复更新 state，否则会导致跳天。

### 微信限流完整修复流程（2026-06-02 验证）
1. 清除 stale context_token：`~/.hermes/weixin/accounts/<account_id>@im.bot.context-tokens.json` 写入 `{}`
2. 杀掉现有 gateway 进程
3. 使用**全局** `Hermes_Gateway.cmd` 重启（非项目 hermes.bat，因项目目录没有微信配置）
4. 等待 10s 验证 `gateway_state.json` 中 weixin: connected
5. Gateway 重新认证获取新 token
**更新规则（必须执行）**：
- 每次推送完成后必须更新进度文件
- current_day 必须 +1
- history 必须追加本次记录
- 如果 current_day > 60，必须重置为 1
- 使用 `execute_code` 而非 `write_file`，因为中文字段名在 write_file 中可能编码异常
